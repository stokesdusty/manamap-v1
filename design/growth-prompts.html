01

Design

mostly apps/mobile

Hierarchy and restraint on screens that already exist. These are front-end-only — no schema, no API. Each names the prototype file and the real screen to change.

G1
Home hierarchy — one hero action Quick win
Home stacks 5+ equal-weight CTAs (Open to Play, Log a Game, Meet Players, Connections, Form a Pod). Give the screen a single state-driven hero and demote the rest.
touches: apps/mobile/src/screens/HomeScreen.tsx · prototype app/screens-home.jsx · theme tokens
G1 · build prompt
Deliverables
  • State-driven hero: checked-in → Open-to-Play/Form-a-Pod; not → Find a store
  • Secondary actions collapse to a compact 2-col grid
  • Single most-urgent item as inline banner (not a list)
  • Prototype app/screens-home.jsx updated to match
Acceptance
  • One clear primary action above the fold per state
  • No two CTAs share the same visual weight
  • Checked-out state points to the store finder
G2
De-dupe "Needs Attention" vs the Connect badge Quick win
Pending confirms + requests show on Home and again as the Connect tab badge. Make Connect the source of truth; Home shows only the single most time-sensitive item.
touches: apps/mobile/src/screens/HomeScreen.tsx · ConnectScreen.tsx · games + connections hooks
G2 · build prompt
Deliverables
  • Home surfaces only the top-priority action item as one banner
  • Connect tab badge stays the canonical count
  • Tapping the Home banner deep-links to the exact item
Acceptance
  • No duplicated standing list of pending items
  • Badge count and Home banner never disagree
G3
Calm the socials row Quick win
The You-tab socials rail is a rainbow of full-saturation brand chips — the loudest thing on an otherwise restrained profile. Default to monochrome glyphs; reveal brand color only on press/active.
touches: apps/mobile/src/screens/YouScreen.tsx · SocialsCard · prototype app/socials.jsx · SOCIAL_META
G3 · build prompt
Deliverables
  • Monochrome glyph tiles by default (neutral chip bg)
  • Brand color on press/active only, or as a thin accent
  • Mana identity remains the profile's color story
  • Prototype app/socials.jsx updated
Acceptance
  • Profile reads calm; identity gradient is the focal color
  • Platforms still recognizable; copy/open unchanged
G4
Reward modal polish Quick win
The "🎁 You unlocked a reward!" modal uses an emoji in an app full of crafted icons — and it's the most commercially important moment. Replace with a branded reward glyph in the store's color.
touches: apps/mobile/src/screens/StoresScreen.tsx (reward modal) · prototype app/screens-stores.jsx
G4 · build prompt
Deliverables
  • Custom reward glyph (react-native-svg) replaces the emoji
  • Store accent color drives the celebration
  • Claim-code + QR state gets matching polish
Acceptance
  • No emoji in the reward / redemption flow
  • Modal matches the app's icon + type system
02

User experience

api + mobile

Close the gap between matching and actually playing, and keep coordination inside the app. G6 is the signature one — the trust layer no competitor has.

G5
Unify the "Play tonight" funnel
"Open to play," LFG, "Form a Pod," and "Meet players" are four overlapping doors to one goal: get into a game tonight. Collapse them under a single entry that branches.
touches: HomeScreen + DiscoverScreen (mobile) · existing lfg + pods modules (no new backend) · prototype app/lfg.jsx, app/pods.jsx
G5 · build prompt
Deliverables
  • One "Play tonight" entry → branch: broadcast open · join pod · start pod
  • Remove the duplicate Open-to-Play surfaces
  • Reuses existing LFG + pods endpoints unchanged
Acceptance
  • A new user reaches a game in one obvious funnel
  • No conceptual overlap between LFG / pods / "open"
G6
Pod Agreement / Rule 0 card Signature
Power level and vibe live on the profile, but the pre-game expectation-setting — bracket, combos, "I'm on my janky deck" — happens nowhere. Add a lightweight table-consensus card when a pod fills. The trust layer no competitor has.
touches: Redis pod sessions (pods.service.ts PodSession) · PodScreen.tsx · prototype app/pods.jsx · shared zod
G6 · build prompt
Deliverables
  • Per-member bracket pick (1–4 / cEDH) + a few flags on the pod session
  • Shared consensus view shown before shuffle-up
  • Bracket data feeds back into power-matching
  • Stored on the Redis PodSession (no Prisma change)
Acceptance
  • Every seat sees the table's agreed bracket + flags
  • Mismatch is visible before the game, not after
  • Expires with the pod TTL; no orphaned state
G7
Minimal pod & connection threads
The moment two players connect, coordination leaves for Discord and never returns. Add a tiny in-app thread (not a chat product) so "running 20 late / meet at table 4 / here's my list" stays in manamap.
touches: new Message model + thread on Connection / pod session · presence WS (ws-adapter, life-tracker pattern) · push via sendPushToUser
G7 · build prompt
Deliverables
  • Message model (threadId, senderId, body, createdAt)
  • Thread per accepted Connection + per live pod
  • Send/list endpoints + WS push for new messages
  • Light thread UI from connection detail + PodScreen
Acceptance
  • Only accepted connections / pod members can post
  • Blocked/moderated users can't message
  • New message pushes + deep-links to the thread
G8
Auto-confirm game results + a nudge
Unconfirmed GameLogs leave stats permanently incomplete and erode trust in the leaderboard. Auto-confirm pending games after 48h (disputable within 7 days), with one gentle push.
touches: GameLog/GamePlayer (GameStatus) · games module · BullMQ (gamification queue pattern) · push
G8 · build prompt
Deliverables
  • Delayed BullMQ job auto-confirms PENDING games at 48h
  • Single reminder push before auto-confirm
  • 7-day dispute window flips status back + recomputes stats
Acceptance
  • No game sits PENDING forever; stats settle
  • Auto-confirmed games are clearly labeled + disputable
  • Dispute within 7 days reverses cleanly
G9
Empty / low-density states as invitations
The seeded demo hides the real cold-start: empty store, no one nearby on a Tuesday. Design the low-density states deliberately — "be the first," one-tap broadcast, "ping me when 2+ check in."
touches: DiscoverScreen + HomeScreen empty states · presence + a new "notify when N here" subscription · push
G9 · build prompt
Deliverables
  • Designed empty states for Discover / Nearby / Home
  • "Be the first — go Open to Play" one-tap CTA
  • "Ping me when 2+ players check in here" subscription + push
Acceptance
  • An empty store feels like an invite, not a dead end
  • The ping fires once when the threshold is crossed
03

New features

full-stack

High-leverage additions that either close the play loop or open a revenue line. Built on data models you already have (DeckLink, GameLog, Encounter, Streak).

G10
Deck shelf + affiliate buy-links Revenue
Players already link Moxfield/Archidekt decks (DeckLink). Make decks first-class: a deck shelf on the profile, per-deck win rate (already derivable from GameLog), and affiliate buy-links for cards — the cleanest path to transaction revenue.
touches: DeckLink model · me/socials-style CRUD · GamePlayer.deck for win-rate join · TCGplayer/Card Kingdom affiliate
G10 · build prompt
Deliverables
  • Deck shelf on the profile (DeckLink, with cover/colors)
  • Per-deck win rate joined from GamePlayer.deck + GameLog
  • Affiliate-tagged buy-links (config-driven partner + tag)
  • Click tracking for affiliate attribution
Acceptance
  • Decks render with real win rate where games exist
  • Buy-links carry the affiliate tag; clicks are logged
  • No affiliate code hard-coded in the bundle
G11
Travel / visiting mode Plus driver
"I'm in a new city this weekend — who's playing, which store, can I get into a pod?" The location graph already exists; this is mostly a mode flip over discovery + a great reason to pay for Plus.
touches: discovery module (PostGIS nearby) · presence · a "visiting {city}" override on discovery queries · gated by Plus (see G16)
G11 · build prompt
Deliverables
  • "Visiting" toggle: set a temporary location/store
  • Discovery + stores + nearby resolve against it
  • Expires automatically; clearly labeled while active
  • Entitlement check (Plus) with a graceful upsell
Acceptance
  • A visiting user sees the destination's stores + players
  • GPS check-in still gates the real reward-grade actions
  • Mode auto-clears; home location is restored
G12
Post-game endorsements
After a confirmed game, optional one-tap positive endorsements ("great host," "good sport," "taught me the format," "fast player"). A positive-reputation signal that improves matching and complements moderation. Strictly positive — no negative reviews.
touches: GameLog confirm flow · new Endorsement model (fromUser, toUser, gameId, tag) · profile aggregate · respects blocks
G12 · build prompt
Deliverables
  • Endorsement model + fixed positive-only tag set
  • Prompt after a confirmed game; one per player per game
  • Aggregated counts on the profile / connection detail
Acceptance
  • Only co-players of a confirmed game can endorse
  • No free-text, no negative options, no review-bombing
  • Blocked users can't endorse; counts respect privacy
G13
Store leagues & seasons (owner-run) Portal premium
Extend Log-a-Game into multi-week store leagues with standings — a recurring reason to check in, and a premium store-portal feature. The Encounter / GameLog model is already there; this is a season construct + standings.
touches: new League + LeagueStanding models · GameLog tagged with leagueId · partner portal (owner-guarded) · mobile standings view
G13 · build prompt
Deliverables
  • League + season model owned by a store (StoreOwnership)
  • Games logged "in league" feed standings (W/L/points)
  • Portal CRUD for leagues; mobile standings + my rank
  • Gated to paid partner tiers (see G14)
Acceptance
  • Owner runs a season; standings update on game confirm
  • Players see the league + their rank at that store
  • Only entitled stores can create leagues
04

Monetization

stores first, then players

Three revenue lines, sequenced so each funds the next. Build G14 + G15 first (stores already pay for foot traffic); G16 (player Plus) lands once density is real. Keep the core free forever.

G14
Store partner tiers (SaaS entitlements) Primary
Your partner portal + reward offers are the wedge. Add subscription tiers that gate the high-value owner tools (broadcast, event mgmt, analytics, leagues, "verified store" badge) behind a paid plan, billed via Stripe.
touches: Store.isPartner · StoreOwnership · new StoreSubscription (tier, status, stripeCustomerId) · Stripe webhooks · entitlement guard on /v1/partner/*
G14 · build prompt
Deliverables
  • StoreSubscription model + tier enum (Free / Plus / Pro)
  • Stripe Checkout + customer portal + webhook sync
  • Entitlement guard gating broadcast / events / leagues / analytics
  • Portal billing page + "verified store" badge for paid tiers
Acceptance
  • Owner subscribes; entitlements unlock immediately on webhook
  • Downgrade/cancel re-locks gated tools at period end
  • Free tier keeps check-in + basic presence (never paywalled)
G15
Sponsored placement & promoted events Primary
Let stores pay to be featured in Discover ("Featured tonight") or boost an event to nearby players. Native to existing surfaces, high margin, scales with store density.
touches: new Promotion model (storeId/eventId, kind, window, budget) · discovery + stores feed insertion · partner portal · clearly-labeled "Featured"
G15 · build prompt
Deliverables
  • Promotion model + active-window resolution
  • Featured store / boosted event slot in Discover + stores
  • Portal: buy a boost (date window, audience radius)
  • Impression/tap logging for billing + reporting
Acceptance
  • Featured items are clearly labeled "Featured" / "Promoted"
  • Only one sponsored slot per feed view; never spammy
  • Impressions/taps logged; respects entitlements (G14)
G16
manamap Plus (player subscription) Secondary
A light consumer tier once density is real: travel mode (G11), advanced stats & matchup analytics, custom card themes/foils, "who viewed your card," unlimited LFG, priority pod seats. Core stays free forever.
touches: new UserSubscription (RevenueCat/StoreKit + Play Billing) · entitlement helper · gate Plus-only features · You-tab paywall
G16 · build prompt
Deliverables
  • UserSubscription model + RevenueCat (IAP) integration
  • isPlus entitlement helper used across gated features
  • Plus features: travel mode, advanced stats, card themes, card-view list
  • You-tab paywall + restore purchases
Acceptance
  • Purchase unlocks entitlements on both platforms; restore works
  • Core check-in / connect / safety never gated
  • Gated features upsell gracefully when locked
